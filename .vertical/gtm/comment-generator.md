# Comment Generator

Use this to write replies under X / Twitter posts. The goal is not to promote
Vertical directly. The goal is to become visible as someone thinking clearly
about coding agents, GTM, and agentic businesses.

## Daily Cadence

Minimum effective dose:
- 2 sessions per day.
- 15 minutes each.
- 5-8 thoughtful replies per session.
- Total: 10-16 replies per day.

Good days:
- 20 thoughtful replies.

Do not do generic replies. If a reply does not add a thought, skip it.

## Target Posts

Prioritize posts about:
- Claude Code, Codex, Cursor, Lovable, Devin, coding agents.
- AI agents moving from demos to real workflows.
- Software factories and agentic engineering.
- Founder-led GTM, sales, distribution, cold outreach, content.
- Indie hackers or technical founders struggling with growth.
- CRM, business memory, agent integrations, tool fragmentation.
- AI-native company building.

Avoid:
- Generic AI hype threads.
- Engagement bait.
- Political/culture-war posts.
- Posts where the only possible reply is "great point."
- Threads where mentioning Vertical would feel forced.

## Target People

Look for:
- Founders building with coding agents.
- Developer tool founders.
- AI agent founders.
- GTM automation founders.
- Indie hackers with shipped products.
- Investors/operators discussing AI-native companies.
- Researchers writing about agents, evals, memory, or tool use.

The best targets are not always huge accounts. Prefer people whose audience
contains builders and founders you want to know.

## Reply Principles

Good replies:
- Add a concrete angle.
- Name a pattern.
- Connect the post to feedback loops, context, distribution, trust, or glue.
- Feel like a working note.
- Can stand alone if someone sees it out of context.

Bad replies:
- "Agree."
- "Great post."
- "We are building this."
- Generic advice.
- Overly polished thought leadership.

## Style Calibration

Reference examples live in `.vertical/gtm/style-reference.md`.

Do not overcorrect into vagueness. The reply should still add a point.

Prefer curious precision over finished lessons:
- "I keep coming back to..."
- "One thing this makes me wonder..."
- "A distinction I've been playing with..."
- "Maybe the interesting part is..."

Translate internal category language into practical language.

Internal:
- distribution
- outcome-based learning
- business workflows
- agentic leverage

External:
- finding people
- earning attention
- did they reply?
- did they buy?
- what changed after?
- building is addictive

When possible, ground replies in a concrete behavior or example before naming the pattern.

Mention Vertical only when:
- Someone asks what you are building.
- The post is directly about the exact problem.
- You can mention it as context, not a pitch.

## Reply Buckets

### The Loop
Use when someone discusses agents, software quality, evals, autonomy, or company building.

Template:
```text
I think the missing piece is the feedback loop.

Coding agents can write code, run tests, see what broke, and try again.

Most business workflows do not have that same loop yet.
```

Variant:
```text
The interesting part to me is not whether the agent can do the task once.

It is whether the system can learn from what happened after.
```

### The Bottleneck Shift
Use when someone talks about shipping faster, indie hacking, SaaS building, or coding agents.

Template:
```text
This feels right.

Building is becoming less of a bottleneck for technical founders.

The harder part is increasingly:

who wants this?
how do we reach them?
what did we learn from the market?
```

Variant:
```text
The funny thing about coding agents is that they make it easier to avoid the hard part.

You can keep building.

But the market still does not care until distribution starts working.
```

### Business Memory
Use when someone discusses CRM, context, memory, agents, customer data, or operating systems.

Template:
```text
I keep coming back to this.

For business agents, memory is not a nice-to-have.

The business is the memory:

customers, conversations, experiments, revenue, product decisions, tasks.
```

Variant:
```text
A lot of "agent memory" discussion feels too abstract.

For a company, the useful memory is very concrete:

what customers said
what we tried
what worked
what changed
what we should do next
```

### The Glue
Use when someone talks about tools, integrations, workflows, APIs, permissions, or agent execution.

Template:
```text
This is where the glue becomes the product.

The business lives across email, CRM, GitHub, Stripe, docs, analytics, and spreadsheets.

An agent that cannot cross those boundaries is mostly an advisor.
```

Variant:
```text
Integrations are easy to dismiss as plumbing.

But for business agents, they define what the system can actually do.

No access to the outside world, no real operation.
```

### Agent Trust
Use when someone discusses UI, autonomy, permissions, human-in-the-loop, or agent reliability.

Template:
```text
One thing I underestimated:

Even in agent-first products, people still want a UI.

Not because they want buttons.

Because they need to understand what happened, why, and what happens next.
```

Variant:
```text
Trust in agents might be less about perfect autonomy and more about legibility.

What happened?
Why did it happen?
What is the system doing next?
Where should I intervene?
```

### Field Notes
Use when someone describes a founder/product behavior you have seen directly.

Template:
```text
We saw a version of this with early users.

Given more agentic leverage, people did not immediately do more distribution.

They mostly kept tweaking products and websites.

The temptation to build is very strong.
```

Variant:
```text
This matches what we have been seeing.

The sharper customer is not "I have an idea."

It is "I built something, have some signal, and now need customers."
```

### Second-Time Founder Notes
Use sparingly when the post is about founders, markets, fundraising, product taste, or iteration.

Template:
```text
The second time around, I care much less about the first idea.

I care much more about the market, the learning loop, and how quickly reality pushes back.
```

Variant:
```text
One painful lesson from building before:

Product quality can hide market quality for a long time.

You can be "winning" and still be in a market that is not worth winning.
```

## Fast Reply Process

1. Read the post.
2. Identify which bucket it touches.
3. Choose one template.
4. Replace abstractions with the specific topic from the post.
5. Delete anything that sounds like a pitch.
6. Post only if it adds something.

## Optional Soft CTA

Use rarely.

```text
This is close to the question we have been exploring with Vertical:

what would it take for agents to not just do business tasks, but learn from business outcomes?
```

Only use this when it is directly relevant.
