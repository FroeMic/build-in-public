# Style Reference

Use this as a calibration file when planning posts, drafting comments, or
rewriting content. The purpose is not to copy these examples. The purpose is to
remember the shape, pacing, and level of abstraction that feels right.

## Calibrated Vertical Drafts

### 1. Building Got Faster

```text
Coding agents made building much faster.

They did not make the market faster.

You can get from idea to working product in days.

But you still have to:

Find people.
Earn attention.
Understand what they care about.
Learn why they buy.
```

Why it works:
- Concrete.
- No forced final lesson.
- Translates "distribution" into lived language.

### 2. Building Is No Longer The Slowest Part

```text
I think something changed for technical founders.

Building is no longer the slowest part.

Coding agents do not make product building easy.

But they do make it much faster to get from idea to working software.

That creates a strange problem:

You can now build faster than you can learn from the market.

You can ship another feature.
Rewrite the landing page.
Improve onboarding.
Launch another small product.

But the harder questions are still there:

Who wants this?
How do we reach them?
What did they understand?
What did they ignore?
Why did they buy?

The next layer of leverage probably isn't more software creation.

It is the loop after software creation:

Product -> market -> learning -> product.
```

Why it works:
- Expands the same idea for LinkedIn without becoming a manifesto.
- Keeps the practical customer questions visible.

### 3. Business Agents: Two Problems

```text
Thinking about business agents today.

It feels like there are two very different problems:

1. getting the agent to do the thing
2. getting the system to learn from what happened after

The second one has layers.

Narrow:
how do we improve the next email?

Broader:
what did this teach us about the customer, product, positioning, and next action?

The first one feels like workflow automation.

The second one feels closer to building a company.
```

Why it works:
- Curious instead of definitive.
- Practical example first, broader thesis second.
- Hints at "The Glue" without naming it.

### 4. Building Is Addictive

```text
One thing that surprised me:

We gave people an agent that would start a business for them.

We expected them to launch within a day.

Most spent their time iterating on products and landing pages.

Not finding customers.

Building is addictive.
```

Why it works:
- Real field note.
- Specific setup and surprise.
- Sharp final line.

### 5. Agentic Engineering Trap

```text
Agentic engineering is both magical and a trap.

Magical because you can move incredibly fast.

A rough idea can become a working product in days.

The trap is that speed feels like progress.

It becomes very easy to build fast things no one uses.

Make it work.
Make it right.
Make it fast.

Agentic engineering makes it much easier to do this in the wrong order.
```

Why it works:
- Sharp hook.
- Concrete danger.
- Uses the internal operating principle naturally.

### Context Engineering Metaphor

```text
A context engineering metaphor I've been playing around with:

- Primary source: the source of truth. Raw data. Transcripts. Code.
- Secondary source: one step removed. Summaries. Compactions. Documentation.

For instance, compaction takes a primary source (the conversation history) and turns it into a secondary source (the summary). This is lossy, but means the secondary source can fit into a smaller space.

If you want to know what your codebase does, your code is a primary source. Your docs are a secondary source.

Loading primary sources into context is expensive, but provides richer context. Secondary sources are cheaper to load into context, but may be information-lossy.

Any context engineering will involve managing the tradeoffs between both.
```

### Metaphor Being Played With

Shape:
```text
A [domain] metaphor I've been playing around with:

- Term one: simple definition.
- Term two: simple definition.

Example.

Tradeoff.

```

What to borrow:
- Starts with "playing around with," not a hard thesis.
- Defines a distinction.
- Uses examples to make the distinction practical.
- Ends by naming the tradeoff, not overclaiming.

## Drafting Heuristics

When a draft feels wrong, check for these failure modes:

- It sounds like a strategy bucket instead of a person thinking.
- It explains the lesson too early.
- It uses internal words where lived language would be better.
- It ends with a generic conclusion.
- It pitches Vertical instead of showing the problem.
- It sounds like polished marketing.
- It asks people to self-identify as leads.
- It uses needy engagement bait like "curious who else is running into this."

Better moves:

- Start with "Thinking about..."
- Start with "A distinction I've been playing with..."
- Start with "One thing that surprised me..."
- Use a concrete example.
- End on the sharpest line, or do not add an ending.
- Say what you are building and let people pull.
